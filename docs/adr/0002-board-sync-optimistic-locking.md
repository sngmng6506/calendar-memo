# ADR-0002: 보드 동기화에 버전 기반 낙관적 잠금을 쓴다

- 상태: 채택(Accepted)
- 날짜: 2026-07

## 맥락

기존 동기화는 로드/로그인 시 **무조건 서버 보드로 로컬을 덮어썼다**(`applyServerBoard`).
free 플랜은 `autoSync: false`라 편집이 자동으로 서버에 올라가지 않으므로, 다음의 흔한
순서에서 데이터가 유실됐다:

1. 사용자가 편집 (로컬·localStorage에만 반영, 서버 push 안 됨)
2. 수동 동기화를 누르지 않음
3. 새로고침 → `fetchServerBoard` → **오래된 서버 보드가 로컬 편집을 덮어씀** → 편집 유실

`syncAfterLogin`의 주석은 "병합"이라 되어 있었지만 실제 코드는 교체였다. 서버 `PUT`도
무조건 last-write-wins이라 두 기기가 동시에 쓰면 조용히 덮였다.

## 검토한 대안

1. **필드 단위 병합 / CRDT** — 동시 편집까지 무손실. 그러나 보드 구조가 중첩 객체(tasks →
   entries, energy 맵)라 병합 규칙이 복잡하고, 개인 프로젝트·수동 동기화 단계에선 과설계.
2. **서버 항상 우선** — 현재 동작. 로컬 편집 유실이 바로 그 버그.
3. **로컬 항상 우선** — 다른 기기의 최신 편집을 잃음. 대칭적으로 나쁨.
4. **보드 단위 버전 비교 + 낙관적 잠금** — 각 보드에 `updatedAt`을 두고 최신 쪽을 채택,
   저장 시 `baseUpdatedAt`으로 충돌(409) 감지. 구현이 가볍고 흔한 유실을 막는다. ← 채택

## 결정

- 보드에 `updatedAt`(epoch ms) 필드를 둔다. 로컬 편집마다 `saveState`에서 갱신.
- DB `boards.updated_at` 컬럼을 서버 버전의 권위로 삼고, `GET /api/board`가 이를 내려준다.
- 프론트는 로드/로그인/충돌 시 `reconcileWithServer`로 로컬·서버 중 `updatedAt`이 큰 쪽을 채택.
- `PUT /api/board`는 `baseUpdatedAt`을 받아 서버 현재 버전이 더 새로우면 `409` + 서버 최신본을
  반환하고 저장하지 않는다. 클라이언트는 그 응답으로 재조정한다.
- 기존 로컬 데이터(updatedAt 없음)는 `normalizeBoard`에서 `0`으로 기본화 → 항상 서버가 우선됨.

## 결과

- "편집 후 새로고침 → 롤백" 버그 해소. 5개 시나리오(로컬 최신 / 서버 최신 / 동률 / 레거시
  로컬 / 서버 시간 null) 격리 테스트로 검증.
- prod에서 `JWT_SECRET` 미설정 시 기동 실패 가드를 함께 넣었다(무관하나 같은 보안 정리 묶음).

## 한계 (남김)

- **보드 단위**이므로 두 기기가 서로 다른 목표를 동시에 편집하면 진 쪽 편집이 유실될 수 있다.
- free 플랜은 수동 동기화라 이 창이 좁다. **auto-sync를 유료 기능으로 켤 때** 대안 1(필드
  병합/CRDT)을 다시 판단한다. 그 시점에 이 ADR을 대체하는 새 ADR을 작성할 것.
