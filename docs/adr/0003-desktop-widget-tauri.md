# ADR-0003: 데스크톱 위젯을 Tauri로 만든다

- 상태: 채택(Accepted) — 결정만. 구현 미착수.
- 날짜: 2026-07
- 관련: [`../WIDGET_PLAN.md`](../WIDGET_PLAN.md)

## 맥락

WIDGET_PLAN의 목표대로 스케줄 보드를 **데스크톱 위젯**으로 확장한다. 위젯은 웹앱을
대체하지 않는 **얇은 read/quick-write 클라이언트**로, 같은 서버 데이터(`/api/widget/*`)를
쓰고 보드 계산 로직을 복제하지 않는다(AGENTS.md 불변식). 프론트는 현재 빌드 스텝 없는
Vanilla JS라 어떤 셸이든 거의 그대로 이식 가능하다.

용도가 "항상 떠 있는 작은 위젯"이므로 **리소스 풋프린트(메모리·배포 크기)가 곧 체감 품질**이다.

## 검토한 대안

| 후보 | 장점 | 단점 |
|------|------|------|
| **Electron** | caffeine-Flow에서 이미 사용 → 학습 비용 0. `widget.html` 거의 그대로 탑재 | 상주 위젯치고 무거움(메모리 150MB+, 배포 80MB+). 용도와 무게 불일치 |
| **Tauri** | 시스템 webview 사용 → 메모리 ~10MB, 배포 ~5MB. Vanilla 프론트 그대로 들어감 | Rust 툴체인 추가. 창 설정 보일러플레이트 |
| **PWA/브라우저 탭** | 구현 최소 | 데스크톱 위젯 느낌(frameless·always-on-top·투명) 불가 |

## 결정

**Tauri**를 채택한다. 상주 위젯의 풋프린트가 사용자 체감 품질에 직결되고, 프론트가
Vanilla JS라 이식 비용이 낮아 Tauri의 유일한 비용(Rust 툴체인)을 상쇄한다.

- 프론트는 현 `widget.html/js/css`를 재사용한다. 데스크톱 전용 로직(창 제어 등)은 얇게 얹는다.
- 창은 frameless · always-on-top · 투명 배경 스타일의 위젯 셸을 목표로 한다(세부는 구현 시).
- **빠른 검증이 필요하면** Electron으로 원형을 먼저 만들어도 무방하다 — 프론트 코드는
  동일하므로 나중에 Tauri로 옮겨도 재작성이 아니다.

## 미해결 (다음 결정 대상)

1. **위젯 인증** — 현재 인증은 httpOnly 세션 쿠키 기반인데, 데스크톱 셸에서 Google GIS 로그인
   + 쿠키 유지가 브라우저만큼 매끄럽지 않다. 위젯은 read 중심이므로 **위젯 전용 장기 토큰/키를
   발급하는 별도 엔드포인트**(예: `/api/widget/token`)를 두어 웹 세션과 분리하는 방향이 유력.
   → 별도 ADR로 확정한다.
2. **자동 새로고침** — `widgetAutoRefresh`는 유료 기능(서버가 진실의 원천). free는 수동/저빈도
   폴링. 폴링 주기·백오프 정책은 인증 결정 이후.
3. **quick-write 범위** — 위젯에서 어디까지 쓰기를 허용할지(예: 컨디션 기록만). 쓰기가 들어오면
   ADR-0002의 보드 단위 동기화 창이 넓어지므로 범위를 좁게 시작한다.

## 결과

- 위젯 셸이 가벼워 상주 사용에 적합. 프론트 재사용으로 중복 없음.
- Rust 툴체인이 CI/빌드에 추가된다. 그 절차는 이 ADR 범위 밖(구현 착수 시 문서화).
